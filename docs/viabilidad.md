# Viabilidad técnica del Mapa de patrones de scroll para _"VibeCheck - La app de bienestar digital para adolescentes (13-18)"_

## TL;DR
- **La suposición central de la spec es correcta a medias y debe matizarse**: en **Android** sí se puede clasificar el uso "por app completa" (Instagram, TikTok, YouTube como unidades) con `UsageStatsManager` + el permiso `PACKAGE_USAGE_STATS`; pero en **iOS no existe** una API pública que entregue datos crudos de uso a una app de auto-conocimiento, y **ninguna red social** ofrece una API en tiempo real para leer cuánto tiempo o qué contenido consume el usuario.
- **No hay granularidad de "tipo de contenido" dentro de una app** (Reels vs Stories vs DMs) por ninguna vía estándar del SO: tanto Android como iOS miden únicamente a nivel de app/paquete completo. Inferir contenido intra-app solo es posible con `AccessibilityService` en Android, lo que conlleva alto riesgo de rechazo y de baneo en Google Play.
- **Enfoque más viable y "creíble" para el prototipo de formación**: Android `UsageStatsManager` real (tiempo por app, desbloqueos, franjas horarias) + clasificación de apps por nombre de paquete como *proxy* de "tipo de plataforma", complementado opcionalmente con **importación manual de exports GDPR** (Instagram "Descarga tu información" JSON, TikTok "Download your data" JSON) como fuente de datos reales históricos, dejando explícito que no hay granularidad intra-app ni tiempo real.

## Key Findings

### 1. Android: SÍ es viable a nivel de app completa, con permiso especial
- `UsageStatsManager` (introducido en API 21 / Android 5.0 Lollipop) devuelve tiempo en *foreground* por paquete, agregado por intervalos. La documentación de Android especifica que los datos se agregan en "days, weeks, months, and years", y en la práctica se puede recuperar uso hasta **7 días** con intervalo diario, **~4 semanas** semanal, **6 meses** mensual y **2 años** anual; para `queryEvents`, la propia doc advierte que "Events are only kept by the system for a few days" (retención de eventos muy corta).
- Vía `queryEvents` / `UsageEvents.Event` se obtienen eventos con *timestamp*: `ACTIVITY_RESUMED`/`ACTIVITY_PAUSED` (foreground/background), `SCREEN_INTERACTIVE`/`SCREEN_NON_INTERACTIVE`, `KEYGUARD_SHOWN`/`KEYGUARD_HIDDEN` (bloqueo/desbloqueo → conteo de desbloqueos), `NOTIFICATION_SEEN` (código 10) y `NOTIFICATION_INTERRUPTION` (código 12). Con esto se reconstruyen sesiones, desbloqueos y franjas horarias.
- **Permiso**: `PACKAGE_USAGE_STATS` es de nivel sistema ("special access"). La nota literal de la documentación oficial es: *"Most methods on this API require the permission android.permission.PACKAGE_USAGE_STATS. However, declaring the permission implies intention to use the API and the user of the device still needs to grant permission through the Settings application. See android.provider.Settings#ACTION_USAGE_ACCESS_SETTINGS."* Es decir: no se concede en *runtime*; el usuario debe activarlo manualmente en Ajustes vía `Settings.ACTION_USAGE_ACCESS_SETTINGS`.
- **Política Google Play**: el Usage Access está permitido pero cae bajo la política "Permissions and APIs that Access Sensitive Information": el permiso debe ser necesario para la funcionalidad núcleo promocionada, con divulgación prominente y consentimiento afirmativo del usuario.
- **Limitación clave (refuta parte de la spec)**: NO desglosa el uso por tipo de contenido dentro de la app. No distingue Reels de Stories de DMs dentro de Instagram. Solo llega a nivel de paquete + clase de actividad (`getClassName`), insuficiente para inferir tipo de contenido de forma fiable.

### 2. iOS: NO hay API pública equivalente para auto-analítica en el propio dispositivo
- Apple expone la Screen Time API (iOS 15/16) mediante tres frameworks: `FamilyControls`, `ManagedSettings` y `DeviceActivity`. Requieren el **entitlement Family Controls**, que es privilegiado y requiere aprobación explícita de Apple para publicar en TestFlight/App Store; está diseñado para control parental.
- Desde **iOS 16** existe la autorización `.individual` (`AuthorizationCenter.shared.requestAuthorization(for: .individual)`), que permite auto-monitorización sin Family Sharing y sin control parental — esta es la vía teórica más cercana al caso de uso de un adolescente monitorizándose a sí mismo. En la WWDC22 Apple lo describió como: *"individual authorization can be used by any number of apps per device… since individual authorization is not for parental control use cases, implicit restrictions for iCloud sign-out and deletion of an app do not apply."* No obstante, esta autorización individual puede ser desactivada trivialmente por el usuario desde los ajustes de la app.
- **Restricción decisiva**: los datos de uso solo son accesibles dentro de una `DeviceActivityReport` extension, fuertemente aislada — sin acceso a red y sin poder mover datos crudos fuera de su *address space*. La app anfitriona **no recibe datos crudos**; solo puede renderizar una vista (SwiftUI) que el sistema rellena. Además, la selección de apps se hace vía `FamilyActivityPicker`, que devuelve **tokens opacos** no identificables ni desencriptables: la app ni siquiera sabe qué app concreta eligió el usuario.
- **Conclusión iOS**: se puede *mostrar* al usuario un informe visual de su uso, pero NO se puede entregar a la app datos crudos exportables para construir un "mapa" propio (menos aún en servidor), ni obtener granularidad de contenido. Un hilo del Apple Developer Forums documenta que reconstruir datos crudos desde la extensión es posible pero Apple lo trata como comportamiento no soportado.

### 3. APIs de redes sociales: NINGUNA da uso en tiempo real ni consumo de contenido
- **Instagram**: la Basic Display API se cerró definitivamente el **4 de diciembre de 2024**. El anuncio oficial de Meta for Developers (publicado el 4-sep-2024) dice: *"Starting on December 4th, 2024, Instagram Basic Display API will no longer be available… After December 4th, 2024, there will no longer be a set of Instagram APIs for consumer developer apps."* Los desarrolladores tuvieron 90 días para migrar. La sucesora (Instagram API with Instagram Login / Instagram Graph API) da contenido y perfil, **no** historial de uso ni tiempo de scroll. No existe API de "Your Activity"/"tiempo en la app" exportable.
- **TikTok**: la Display API solo devuelve datos del propio usuario autenticado (perfil, sus propios vídeos publicados), sin tiempo de visionado ni acceso a la "For You Page". La Research API es exclusiva para investigadores académicos/sin ánimo de lucro aprobados (cuota ~1.000 peticiones/día, rechazo habitual de usos comerciales). Existe una **Data Portability API** (2024) que es la vía oficial más cercana pero limitada (ver Details).
- **YouTube**: el Data API v3 **no tiene** endpoint de historial de visionado. La propiedad `contentDetails.relatedPlaylists.watchHistory` fue deprecada el 11-ago-2016 y devuelve vacío/placeholder desde entonces; `activities` está deprecado y no hay *scope* de OAuth para historial.
- **Confirmación explícita solicitada**: por el *sandboxing* de iOS/Android y por las políticas de privacidad y de competencia de estas plataformas, **ninguna red social ofrece una API en tiempo real** que permita a una app de terceros leer cuánto tiempo lleva un usuario scrolleando o qué tipo de contenido está consumiendo en ese momento. Esto es una limitación estructural, no un hueco temporal de producto.

### 4. Fuente de datos "reales" más creíble: los exports GDPR/CCPA
- Instagram "Descarga tu información" (formato JSON) y TikTok "Download your data" (JSON, no TXT) permiten al usuario obtener su propio historial, y son la vía oficial y segura (sin ceder contraseñas).
- **Limitación crítica de granularidad temporal**: estos exports dan autor + timestamp (Instagram: "posts viewed"/"videos watched" incluyen solo autor y hora de visionado, sin identificar la pieza exacta) o fecha + link de vídeo (TikTok). **NO incluyen tiempo de visionado/duración por pieza**, por lo que no permiten reconstruir cuánto tiempo se vio cada contenido.

## Details

### Android `UsageStatsManager` — inventario preciso
`UsageStats` contiene: `getPackageName()`, `getTotalTimeInForeground()`, `getLastTimeUsed()` y begin/end del intervalo. Datos concretos para el "mapa":
- **Tiempo por app**: SÍ (`getTotalTimeInForeground`).
- **Número de desbloqueos**: SÍ, inferido de `KEYGUARD_HIDDEN` (código 18, "typically happens when the user unlocks their phone after turning it on") y/o `SCREEN_INTERACTIVE`.
- **Notificaciones recibidas**: PARCIAL. `NOTIFICATION_INTERRUPTION` (código 12) existe pero está marcado `@hide`/`@SystemApi` en el código fuente de AOSP (no oficialmente accesible a apps de terceros); `NOTIFICATION_SEEN` (código 10) está documentado. El conteo fiable de notificaciones para apps de terceros no está garantizado.
- **Franjas horarias de uso**: SÍ, a partir de los timestamps de los eventos.
- **Tipo de contenido intra-app**: NO por esta vía.

Nota sobre nombres de paquete para el proxy de clasificación: Instagram = `com.instagram.android`; TikTok = `com.zhiliaoapp.musically`; YouTube = `com.google.android.youtube`. Estos permiten mapear "app → tipo de plataforma" (social/vídeo corto/vídeo largo) como proxy de "tipo de contenido a nivel de app".

**Digital Wellbeing**: Google **no** expone una API pública para exportar datos de Digital Wellbeing a terceros. Históricamente las apps de terceros solo podían crear horarios de "No molestar" con las APIs de Wellbeing; Android 15 abrió parcialmente funciones (p.ej. Dark Mode, apagar Always-On Display) pero **no** la exportación de datos de uso. La única fuente equivalente para terceros sigue siendo `UsageStatsManager`.

**Riesgo emergente con `AccessibilityService`** (única vía técnica para inferir contenido intra-app en Android): Google Play solo exime de divulgación a apps que declaran `isAccessibilityTool=true` y cuyo propósito primario es asistir a personas con discapacidad; el resto debe completar una declaración con justificación y vídeo, y "using the API for general automation or non-accessibility purposes is not allowed". Además, Android 17 (beta) con Advanced Protection Mode **revoca y bloquea** automáticamente el permiso de accesibilidad a apps no clasificadas como herramientas de accesibilidad. Usar AccessibilityService para analítica de bienestar (no discapacidad) tiene alto riesgo de rechazo/retirada.

### iOS — `FamilyControls` / `ManagedSettings` / `DeviceActivity`
`FamilyControls` es la puerta de autorización; `ManagedSettings` aplica las restricciones (aplicadas por el SO, no por la app); `DeviceActivity` monitoriza (`DeviceActivityMonitor`, eventos por umbral) y renderiza informes (`DeviceActivityReport`). Los informes se filtran por `DeviceActivityFilter` (segmentos `.daily`/`.hourly`, dispositivos, apps/categorías) y exponen agregados como `totalActivityDuration` dentro de la extensión aislada — pero esos datos no pueden salir hacia la app anfitriona ni a la red. Esto es intencional y descrito por Apple como "privacy-preserving".

### TikTok Data Portability API — el hallazgo más relevante para "datos reales"
Documentada en developers.tiktok.com (anunciada en 2024). Estructura de *scopes* OAuth: `portability.<tipo>.single` y `portability.<tipo>.ongoing`, con `<tipo>` ∈ {`all`, `activity`, `directmessages`, `postsandprofile`} (ocho scopes en total). Categorías: "Posts and Profile", "Activity", "Direct Messages" y "Full Archive"; cada una admite una petición puntual (single) o acceso continuado (ongoing).

**Dato decisivo**: dentro de "Activity", la sub-categoría **"Watch History" expone exactamente dos campos — "Watch History Date" y "Video landing page link" — SIN ningún campo de watch time / duración por vídeo.** La única duración de visionado que aparece en el archivo es la de las retransmisiones LIVE propias del usuario ("LIVE Duration: start time - end time (duration mins)"), no la del feed. 

Condiciones de acceso (doc oficial): *"Our Data Portability API, available for TikTok users in the European Economic Area (EEA) or United Kingdom (UK), allows users to authorize a transfer of their information to your app."* Es **aprobada por solicitud** (revisión de privacidad y seguridad + mockups UX de alta fidelidad; respuesta típica en 3-4 semanas), **limitada a usuarios de EEA/UK** (no devuelve datos fuera de esas regiones), requiere consentimiento OAuth del usuario, y es **asíncrona** (flujo: Add Data Request → Check Status → callback vía webhook → Download; los datos quedan disponibles 4 días). No es una API de tiempo real. Se enmarca en "market contestability"/portabilidad de datos (contexto DMA de la UE, aunque el texto usa "market contestability", no cita literalmente el DMA).

### Terceros que resuelven parte del problema (inspiración técnica)
- **RescueTime (Android)** usa exactamente `UsageStatsManager` (permiso "App usage data") para el tracking de tiempo por app, y reserva `AccessibilityService` únicamente para bloquear apps durante sesiones de foco. Su API OAuth2 ofrece datos agregados por categoría y nivel de productividad — no granularidad de contenido. Es el patrón técnico a replicar en Android.
- **RescueTime en iOS** no puede trackear apps/webs por las limitaciones de Apple; recurre a *location tracking*, lo que confirma la ausencia de una vía oficial de datos de uso en iOS.
- **Exports de Screen Time en iOS**: los entusiastas extraen datos de la base `knowledgeC.db` (requiere Mac con misma cuenta iCloud, o jailbreak) o vía Shortcuts + OCR de capturas de pantalla. Ninguna de estas rutas es escalable ni apta para producción o para un demo de formación limpio.

## Recommendations

**Etapa 1 — MVP/demo (recomendado): Android + `UsageStatsManager` real.** Es la única fuente que entrega datos reales de uso por app con permisos estándar del SO. Mapear paquetes conocidos a categorías de "tipo de plataforma" como proxy de "tipo de contenido". Presentar tiempo por app, desbloqueos y franjas horarias. **Corregir la spec** para dejar explícito que la clasificación es a nivel de app completa, no intra-app.

**Etapa 2 — Enriquecer credibilidad con datos reales del usuario: importación manual de exports GDPR.** Añadir un flujo donde el adolescente descarga su Instagram JSON / TikTok JSON y lo sube a la app; aporta historial real de piezas vistas con timestamps (fuente "real" y no simulada para la demo). Advertir en la UI que no incluye duración por pieza ni tipo de contenido dentro de la app.

**Etapa 3 — iOS (solo si es imprescindible): informe visual, no datos crudos.** Planificar exclusivamente un `DeviceActivityReport` con autorización `.individual`, asumiendo que (a) requiere aprobar el entitlement Family Controls con Apple, (b) no se obtienen datos crudos exportables, (c) no hay granularidad intra-app, (d) el usuario puede desactivar la autorización individual.

**Qué NO hacer:** no prometer detección de "Reels vs Stories vs DMs" ni lectura en tiempo real del scroll; no depender de APIs de redes sociales para uso/consumo (no existen); no construir el producto sobre `AccessibilityService` por el riesgo de rechazo/baneo en Google Play y el bloqueo en Android 17.

**Umbrales/benchmarks que cambiarían esta recomendación:** (1) que Apple abra una API de datos de uso accesible fuera de la extensión aislada; (2) que Instagram/TikTok/YouTube publiquen endpoints oficiales de "tiempo en la app"/watch-time por usuario final; (3) que se consiga una colaboración/partnership oficial de plataforma con acceso a granularidad intra-app; (4) que la TikTok Data Portability API añada un campo de duración de visionado o se extienda fuera de EEA/UK. Cualquiera de estos habilitaría granularidad intra-app o tiempo real.

## Caveats
- Las políticas de plataforma cambian con frecuencia; **toda la información está fechada a julio de 2026**. Verificar la documentación oficial antes del desarrollo.
- Los hechos clave (deprecación de Instagram Basic Display el 4-dic-2024, campos de la TikTok Data Portability API, nota de permiso de `UsageStatsManager`, restricciones de `AccessibilityService` y Android 17, aislamiento de `DeviceActivityReport`) están corroborados con documentación oficial de Meta, TikTok, Apple y Google; parte del detalle operativo sobre exports proviene de fuentes secundarias (blogs, foros) y se ha marcado como tal.
- El evento `NOTIFICATION_INTERRUPTION` está marcado como oculto/sistema en AOSP; el conteo fiable de notificaciones para apps de terceros no está garantizado y debe validarse en dispositivos reales.
- La cifra de "10.000–50.000+ entradas" en exports de TikTok que circula en blogs no está respaldada por fuente oficial y se ha omitido deliberadamente.
- Consideraciones legales adicionales para menores (13-18 años): el RGPD y su tratamiento de datos de menores, COPPA (EE. UU.) y las políticas de Google Play para apps dirigidas a niños/adolescentes aplican y requieren revisión jurídica específica, especialmente si se procesan datos de uso en servidor.
